# ARCH Template

## 生成约定

- 首稿应先锁定技术边界、模块职责、数据流、验证方式和主要风险
- 如果需求范围还不稳定，应先收敛范围，再落架构
- 最终内容必须沉淀回本文件，不能只停留在聊天记录
- 未写入本文件的重大技术决策，默认不直接实施
- 回归测试与生产前验证方案，也要在本文件中明确写清触发时机和执行方式
- 多前后端项目默认采用前后端独立目录，不把两端代码混放在同一 `src/`
- 默认启动方式使用 Docker，默认数据库使用 SQLite；若偏离默认值，需写出原因和迁移成本
- 默认开发顺序是先前端后后端，后端切片应建立在已确认的前端边界之上

分层规则：

- `apps/web` 中的界面层只处理展示和交互
- `apps/web` 中的状态层只处理状态流转
- `apps/web` 中的服务层只处理接口通信
- `apps/server` 中按 controller/service/repository 或 feature-first 分层
- 业务规则不要直接散落在页面组件里，也不要混进数据库访问层

## 默认架构约定

### 前后端隔离

- 默认使用 `apps/web` 承载前端代码，`apps/server` 承载后端代码
- 两端构建、依赖、镜像和运行入口相互独立
- 共享约定放在 `reference/`，而不是通过跨目录直接引用业务代码实现“伪共享”

### 启动方式

- 默认使用 Docker 作为本地启动与联调入口
- 文档、脚本和验收说明应优先给出容器化命令
- 若允许裸机启动，只能作为补充路径，不能替代默认路径

### 数据库默认值

- 默认数据库使用 SQLite
- 适用于模板初期、本地开发和小规模验证
- 若切换到 PostgreSQL、MySQL 等，需要明确触发条件、迁移策略和新增运维成本

### 开发顺序

- 先实现并确认前端切片
- 前端阶段先明确页面结构、状态流转、错误反馈和接口契约占位
- 后端阶段再实现 API、业务规则和 SQLite 持久化
- 如果前端需求未稳定就先做后端，最容易造成接口返工；这是默认不推荐方案

### 外部参考索引

- 可在 `reference/agency` 保存外部 agent Markdown 快照与索引，作为本地技能之外的后备知识源
- 该索引的优先级低于本地技能和项目文档
- 若请求已明确命中 `frontend-dev`、`fullstack-dev` 等本地技能，则不再额外加载同领域 agency 文档
- 动态加载的 agency 文档只用于当前任务，不作为长期持久上下文
