# Core Beliefs

这些信念用于约束 agent-first 工作流的长期方向。

## 1. AGENTS.md 是目录，不是百科全书

- 根入口应该短、稳、可维护
- 长规则和领域知识应该放在有结构的 `docs/`
- 如果一个规则需要大量上下文，说明它不该继续留在入口文件里

## 2. 知识不进仓库，就等于不存在

- 聊天记录、口头共识、Slack 讨论不应成为关键依赖
- 任何会影响实现、评审、验证或回滚的知识，都应以版本化文档形式落到仓库

## 3. 计划是一级工件

- 复杂任务先写计划，再写代码
- 计划要写边界、验收、验证和修改范围
- `project_state.md` 只保留摘要，详细执行过程进入 `docs/exec-plans/`

## 4. agent 可读性优先

- 优先选择清晰、稳定、可推理的结构，而不是只对人类短期方便
- 比起一堆临时提醒，更应提供明确的目录、分层、契约和命名

## 5. 能机械化的规则，尽量机械化

- 风格不变量、目录边界、文档完整性、命名约束和计划状态应逐步转为可检查规则
- 文档只是第一步，不是终点

## 6. 技术债要持续清理

- 技术债像高息贷款，越晚处理越贵
- 小额、高频、自动化的清理比大扫除更有效
- 发现反复出现的坏模式时，要么补文档，要么补工具，要么补检查

## 7. 验证是切片完成的一部分

- 没有验证记录的完成不算完成
- 页面流程、日志、指标、回滚条件和失败恢复应能在文档中找到
