# Vibe Coding Workflow Template

本文件是本模板的主入口，不是百科全书。

目标：

- 让 agent 先拿到稳定地图，再按任务读取最相关的知识
- 把长期知识收敛到版本化的 `docs/`，不把关键约束留在聊天记录里
- 用显式计划、切片边界和验证闭环降低 agent 漂移

## 启动顺序

每次开始新会话，先按这个顺序读取：

1. `AGENTS.md`
2. `project_state.md`
3. `docs/index.md`
4. 再根据任务类型读取对应文档，而不是一开始全量读所有文档

如果文档缺失、互相冲突，或当前切片与现有计划不一致，先指出问题，再继续实现。

## 文档路由

- 产品范围、用户价值、成功指标、非目标：`docs/product-specs/`
- 架构地图、模块边界、分层与目录：`ARCHITECTURE.md`
- 设计原则、核心信念、ADR、trade-off：`docs/design-docs/`
- 当前与历史执行计划：`docs/exec-plans/`
- 前端实现和视觉规则：`docs/FRONTEND.md`、`docs/DESIGN.md`
- 产品判断与需求取舍：`docs/PRODUCT_SENSE.md`
- 质量、验证、反熵与技术债：`docs/QUALITY_SCORE.md`、`docs/RELIABILITY.md`
- 安全边界：`docs/SECURITY.md`
- agent 参考资料和外部快照：`docs/references/`

`docs/` 是长期知识的唯一可信来源。聊天记录、口头约定和临时结论不算完成，必须回写到代码库。

## 根目录约定

根目录只保留薄入口和当前状态：

- `AGENTS.md`：导航与执行协议
- `CLAUDE.md`：兼容入口
- `ARCHITECTURE.md`：高层架构地图
- `project_state.md`：当前激活切片、阻塞、最近验证
- `docs/`：全部长期知识

如果需要正式 PRD、设计文档、计划、质量规则或参考资料，统一进入 `docs/`。

## 执行协议

### 1. 先读地图，再动手

- 不允许跳过 `AGENTS.md`、`project_state.md` 和 `docs/index.md`
- 不允许只根据代码表象反推需求
- 如果当前任务找不到对应文档入口，先补文档导航，再推进实现

### 2. 计划优先

- 新任务先确认是否已有对应计划文件
- 中等及以上复杂度任务，必须在 `docs/exec-plans/active/` 建或更新计划
- 计划至少写清：目标、范围边界、不做什么、允许修改的文件、验证方式、完成条件
- 完成后将计划移入 `docs/exec-plans/completed/`

### 3. 一次只做一个可验收切片

- 当前切片必须能独立验证
- 切片必须写清范围边界，例如“只完成登录提交链路，不做找回密码”
- 一个切片里不要混入无关重构、视觉大改和基础设施改造

### 4. 改动范围显式收敛

- 每次任务都要说明允许修改的文件或目录
- 非本次范围文件，除非修阻塞问题，否则不改
- 发现文件承担多个职责时，优先拆分，不继续堆功能

### 5. 文档与代码一起演进

- 需求变化，更新 `docs/product-specs/`
- 架构变化，更新 `ARCHITECTURE.md` 或 `docs/design-docs/`
- 计划推进，更新 `docs/exec-plans/`
- 当前状态变化，更新 `project_state.md`

### 6. 先验证，再结束切片

每个切片完成后至少执行：

1. 按计划或产品规格手动验收
2. 运行 lint
3. 运行类型检查
4. 运行测试
5. 记录验证结果
6. 提交独立 commit

如果项目尚无完整检查链路，要明确写出缺失项，不能假装已验证。

### 7. 能机械化的规则，逐步机械化

- 高频重复的风格、边界和结构规则，应优先沉淀到 linter、脚本、模板或生成产物
- 不要长期依赖“大家记得就好”
- 文档规则如果反复被违反，应升级成可检查约束

### 8. 两次无新证据，不要盲修

遇到问题时按这个顺序：

1. 最小复现
2. 补日志、截图、指标或测试
3. 基于证据缩小范围
4. 必要时写测试固定问题
5. 连续两轮无新证据时，暂停盲修，换策略或回退

## 默认实现边界

- 多应用项目默认使用 `apps/web` 与 `apps/server`
- 前端优先使用 `frontend-dev`
- 后端优先使用 `fullstack-dev`
- 默认开发顺序是先前端、后后端
- 默认启动方式使用 Docker
- 默认数据库使用 SQLite，偏离默认值必须写入文档
- `docs/references/agency` 只作第二优先级后备来源，不能覆盖已命中的本地技能

## 建议开场指令

```text
先读取 AGENTS.md、project_state.md 和 docs/index.md。
根据当前任务选择最相关的 docs 路径，而不是全量读取全部文档。
总结当前目标、当前切片、已知问题、允许修改范围和对应的执行计划。
只在该范围内完成本次任务；如果缺少计划、架构边界或验证方式，先补文档再实现。
完成后更新 project_state.md 和对应的 docs/exec-plans 文件，并记录验证结果。
```
