# Vibe Coding Workflow Template

本文件是本模板的主规则文件。

兼容约定：

- `AGENTS.md` 是主维护入口
- `CLAUDE.md` 是给 Claude 系工具的兼容入口
- 后续若修改任一规则，必须同步更新这两个文件，避免不同入口内容漂移

这个目录是一套可复用的工作流模板，用于把 AI 辅助开发约束在可维护、可验证、可回滚的边界内。

## 目标

- 先定义边界，再让 agent 执行
- 让每次对话都能继承准确上下文，而不是从零猜
- 把需求、架构、状态、规范分开，避免文档和代码职责混乱
- 用小步提交、显式验收和安全底线，降低 AI 失控成本

## 标准目录

```text
vibe-coding-standard/
├── AGENTS.md
├── PRD.md
├── ARCH.md
├── project_state.md
└── reference/
    └── theme.ts
```

## 文件职责

- `PRD.md`
  用户视角的产品需求文档。定义做什么、不做什么、怎么验收。
- `ARCH.md`
  工程视角的约束文档。定义怎么做、边界怎么切、什么不能做。
- `project_state.md`
  当前切片、已知问题和下一步。它不是总结，而是 agent 的短期工作上下文。
- `reference/`
  团队和 agent 需要复用的规则样板，例如主题、接口约定、编码规范。

## 文档生成约定

生成或大改 `PRD.md` 与 `ARCH.md` 时，不要直接自由发挥，优先使用 gstack 里的规划类技能完成首稿，再人工收敛。

- `PRD.md`
  优先使用 `plan-ceo-review` 做范围澄清、目标收敛、版本取舍。
- `ARCH.md`
  优先使用 `plan-eng-review` 做架构边界、数据流、模块职责、验证与风险梳理。

补充约定：

- 如果产品强依赖界面与视觉系统，生成 `PRD.md` 前后可补充使用 `design-consultation`
- 如果是基于现有线上产品补全文档，可补充使用 `plan-design-review`
- 技能产出是输入材料，不替代最终文档责任；最终版本仍要落回 `PRD.md` 与 `ARCH.md`

## 测试与生产前验证约定

测试回归和生产前验证也优先使用 gstack 技能，不只依赖本地命令行检查。

- 回归测试
  优先使用 `qa` 或 `qa-only`
- 生产前烟雾验证
  优先使用 `browse`
- 界面与交互一致性检查
  优先使用 `qa-design-review` 或 `plan-design-review`

执行原则：

- 功能切片完成后的本地验证，仍然要做 lint、类型检查、单元测试
- 面向真实页面流程的回归测试，优先交给 gstack 的 QA 技能执行
- 上线前至少做一轮基于真实入口路径的 smoke test，而不是只看代码和日志
- 若本次改动影响视觉层级、布局、交互反馈，还要补一轮设计向检查

## 对 Agent 的核心约束

### 1. 先读文档，再动代码

每次开始一个新会话，agent 必须先读取：

1. `PRD.md`
2. `ARCH.md`
3. `project_state.md`

原因：

- agent 不具备跨会话稳定记忆
- 如果不先读文档，agent 会用代码表象反推需求，容易越界
- 文档是显式约束，代码只是当前状态，不能替代需求和架构意图

执行要求：

- 第一条工作消息必须明确说明已读取三份文档
- 如果三份文档缺失或互相冲突，先指出冲突，再继续实现
- 不允许跳过文档直接根据口头描述大范围改动

### 2. 用户负责边界和验收，agent 负责实现

职责拆分：

- 用户或产品 owner 决定范围、优先级、验收标准
- agent 负责按约束实现，不擅自扩展需求

这条约束的意义：

- 防止 agent 把“合理推断”演变成“私自加功能”
- 防止需求、设计、实现混在一次对话里失真

执行要求：

- 任何未写入 `PRD.md` 的功能，默认不做
- 任何未写入 `ARCH.md` 的技术改造，默认不做
- 如发现当前需求会突破既定边界，先暂停并说明影响

### 3. 一次只做一个可验收切片

切片定义：

- 一个能独立验证的小功能单元
- 必须可以从用户入口一路打通到最终结果
- 必须有明确完成标准

禁止行为：

- 一次并行推进多个页面和多个后端能力
- 先铺很多页面骨架，最后再接逻辑
- 在一个提交里混入功能、重构、视觉改版和基础设施改造

执行要求：

- 每次任务必须写明“当前切片”
- 切片描述必须包含范围边界，例如“只完成登录提交链路，不做找回密码”
- 完成后必须能手动验证

### 4. 改动范围必须显式收敛

agent 的常见失控方式不是代码写错，而是改得太多。

执行要求：

- 每次任务都要说明本次允许改哪些文件或目录
- 非本次范围文件，除非修复阻塞问题，否则不改
- 发现文件开始承担多个职责时，优先拆分，不继续堆功能

建议约束：

- 单文件接近 300 行时，评估拆分
- UI、状态、数据访问、业务规则不要混在同一文件

### 5. 文档必须与代码同步更新

`project_state.md` 不是周报，而是当前真实状态。

每次切片完成后，agent 必须同步更新三部分：

1. `当前阶段`
2. `已知问题`
3. `下一步`

更新规则：

- 已完成的任务从“当前阶段”移动到“已完成”
- 已修复的问题从“已知问题”移除
- 新发现但未处理的问题加入“已知问题”
- “下一步”必须是真实的下一个切片，不写空话

原因：

- 下一次新会话时，agent 依赖这个文件恢复上下文
- 如果状态文档滞后，AI 会在旧世界观上继续编码

### 6. 先验证，再提交

每个切片完成后，至少执行以下检查：

1. 按 `PRD.md` 手动验收
2. 运行 lint
3. 运行类型检查
4. 运行测试
5. 提交 git commit

这不是形式主义，而是为了拦截三类常见问题：

- 语法和风格错误
- 类型和接口不匹配
- 改坏已有行为

如果项目里还没有完整测试，也必须明确说明缺失项，不能假装已验证。

如果改动已经形成可操作页面或真实用户流，再补充一轮 gstack 验证：

- 回归检查优先用 `qa` 或 `qa-only`
- 生产前 smoke test 优先用 `browse`
- 视觉回归优先用 `qa-design-review`

### 7. 安全约束默认从严

agent 容易优先满足“能跑”，忽略“能守住边界”。

必须遵守：

- 密钥只放服务端环境变量，不写入前端
- `.env` 不入库，使用 `.env.example` 记录变量名
- 服务端接口必须做鉴权，不信任前端参数
- 涉及用户数据的查询必须按当前用户身份过滤
- 登录、验证码、重置密码等敏感接口必须有频率限制

若任务涉及以下内容，必须先在 `ARCH.md` 写清楚再实现：

- 认证方案
- 支付流程
- 数据留存与删除
- 第三方依赖失败后的降级策略

### 8. 两次无新证据，不要盲修

agent 遇到报错时，最容易陷入“换个猜法再试一次”的循环。

正确流程：

1. 最小复现问题
2. 补充日志或观测数据
3. 基于证据缩小范围
4. 必要时写一个小测试固定问题
5. 若连续两轮没有新增证据，停止盲修，换策略或回退到稳定点

这条规则的目标不是保守，而是避免在错误假设上扩大损失。

### 9. Git 是安全带，不是可选项

每个切片完成后都应单独提交。

提交信息要求：

- 描述本次切片完成了什么
- 不使用“misc fixes”这类模糊信息
- 不把多个无关改动塞进一次提交

价值：

- 出问题时可以快速回到稳定点
- 方便定位是哪次改动引入了回归
- 让 agent 的行为历史可审计

### 10. 能用模板解决的，不让 agent 每次重新发明

`reference/` 目录用于收敛重复决策。

适合放进去的内容：

- 颜色、字号、间距、圆角等视觉变量
- API 返回格式约定
- 错误码约定
- 表单校验规则
- 常用组件和网络层封装样板

原则：

- 高频复用的规则，模板化
- 一次决定，多处复用
- 不把临时讨论散落在聊天记录里

## 推荐协作流程

### 开发前

1. 用对话澄清目标、用户、范围和不做什么
2. 写 `PRD.md`
3. 写 `ARCH.md`
4. 建 `reference/`
5. 初始化 `project_state.md`
6. 确认验证命令与提交规则

### 开发中

1. 新会话先读三份文档
2. 明确本次切片范围
3. 限定允许修改的文件
4. 实现切片
5. 手测和运行检查
6. 更新 `project_state.md`
7. 提交 commit

### 开发后

1. 回看未完成项和已知问题
2. 补齐文档和模板沉淀
3. 清理临时方案，避免把权宜之计固化

## 建议给 Agent 的标准开场指令

```text
先读取项目根目录下的 PRD.md、ARCH.md 和 project_state.md。
总结当前目标、当前切片、已知问题和允许修改的文件范围。
只在该范围内完成本次任务，不要顺手重构，不要新增未写入 PRD 的功能。
完成后更新 project_state.md，并说明验证结果。
```

## 什么时候这套约束会失效

以下情况要主动停下来，而不是硬做：

- `PRD.md` 里没有验收标准
- `ARCH.md` 没定义认证、数据边界或部署约束
- `project_state.md` 明显过期
- 当前任务横跨多个切片，无法在一次提交中清楚收敛
- 用户临时口头要求与现有文档冲突，但未决定以哪个为准

## 模板使用原则

- 文档必须放项目根目录，不能埋在子目录里
- 模板是起点，不是替代思考
- 任何新约束，如果会长期复用，应沉淀到模板或 `reference/`
- 这套流程优先追求“能持续演进”，不是追求第一次就写出完美架构
