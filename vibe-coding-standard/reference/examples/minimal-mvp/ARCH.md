# Minimal MVP Example ARCH

## 生成约定

- 本示例用于个人快速验证想法，因此优先选择最容易实现和回退的方案
- 目标是以最少系统复杂度完成“编辑 -> 预览 -> 分享”闭环
- 如果某项复杂度不会提升验证速度，就默认不做

## 技术目标

- 当前阶段：个人验证想法的第一版
- 优先级：最快做出一个可访问页面，而不是做完整后台
- 主要技术风险：做得太复杂、为了未来扩展过早设计、把验证项目做成半成品平台
- 本期主要服务的 PRD 目标：缩短从想法到页面可访问的时间，并快速收集初步反馈
- 本期明确不支撑的 PRD 非目标：多用户、复杂权限、通知、评论、正式 CMS

## 用户范围与规模

- 自用 / 小范围内测 / 公开上线：自用 + 小范围分享
- 预估并发用户数：1 到 5 人同时访问
- 预估数据规模：通常只有少量页面内容
- 峰值场景或流量假设：链接刚分享出去时短时有少量访问

## 架构选型与取舍

### 总体架构模式

- 推荐模式：纯前端或前端 + 极薄后端
- 为什么选它：实现最快，最容易修改和回退
- 当前阶段不选其他模式的原因：数据库、认证、后台管理都会拖慢验证速度
- 未来可能触发演进的信号：需要多人编辑、需要稳定保存大量页面、需要统计访问数据

### 关键技术决策（ADR）

| 决策项 | 可选方案 | 当前选择 | 选择原因 | 代价/放弃了什么 | 何时需要重审 |
| --- | --- | --- | --- | --- | --- |
| 应用形态 | 纯前端 / 前后端分离 | 纯前端优先 | 开发最快，足够支撑个人验证 | 数据能力弱，不适合多人协作 | 需要远程持久化时 |
| 数据存储 | 内存 / localStorage / SQLite | localStorage 或本地文件 | 无需额外部署，最容易起步 | 换设备会丢数据 | 需要跨设备同步时 |
| 发布方式 | 本地运行 / 静态站点 | 静态站点 | 分享成本最低 | 动态能力有限 | 需要服务端接口时 |

## 技术栈

- 客户端：React + TypeScript，或任一熟悉的轻量前端框架
- 服务端：默认无
- 数据库：默认无；如必须使用，则优先 SQLite
- 部署平台：静态托管或本地运行
- 第三方服务：默认无
- 配置与密钥管理：默认无敏感配置

## 运行时拓扑与部署约束

- 运行进程/容器：默认单前端应用
- 服务间依赖顺序：无
- 对外暴露端口：一个前端访问入口即可
- 内部网络与服务发现方式：不适用
- 持久化卷或数据目录：如使用本地文件，则放在项目目录可见位置
- 健康检查方式：首页可访问即可
- 环境变量分层：本期通常不需要
- 备份与恢复策略：导出一份内容 JSON 或手动复制文案
- 第三方服务故障时的降级策略：默认无第三方依赖

分层规则：

- 本项目采用：极简前端分层
- 选择原因：个人验证阶段不需要复杂边界
- 允许例外的条件：文件很小，可以适度合并
- 页面负责流程，组件负责复用片段，数据层负责最小存取
- 一旦单文件明显承载多个职责，再拆分

## 模块边界

| 模块 | 职责 | 不负责什么 |
| --- | --- | --- |
| editor | 输入和校验内容 | 公开渲染 |
| preview | 把当前内容渲染成页面效果 | 持久化逻辑 |
| public_page | 公开展示最终页面 | 编辑逻辑 |
| storage | 保存和读取最小内容 | UI 渲染 |

## 数据模型

| 实体 | 关键字段 | 关系 |
| --- | --- | --- |
| landing_note | `title`, `problem`, `solution`, `cta_text` | 单实体即可支撑 MVP |

补充要求：

- 字段尽量少，只保留验证价值所必需的内容
- 如果某个字段不是验证核心，就先不加

## 核心工作流清单

| 工作流 | 触发入口 | 主要参与方 | 成功结果 | 失败结果 | 当前状态 |
| --- | --- | --- | --- | --- | --- |
| 编辑内容 | 编辑页输入表单 | 用户、Web | 表单内容更新 | 提示缺失字段 | Draft |
| 预览页面 | 编辑页触发预览 | 用户、Web | 立即看到页面效果 | 显示预览失败提示 | Draft |
| 公开访问 | 打开公开页链接 | 访问者、Web | 页面正常显示 | 显示内容不存在 | Draft |

## 工作流状态与失败恢复矩阵

| 工作流/实体 | 状态 | 进入条件 | 用户可见表现 | 系统动作 | 失败/超时后的恢复方式 |
| --- | --- | --- | --- | --- | --- |
| landing_note | `draft` | 开始输入内容 | 编辑页可继续修改 | 保存在内存或本地 | 刷新后从本地恢复或重新输入 |
| landing_note | `preview_ready` | 必填字段齐全 | 预览区域可见 | 渲染页面预览 | 渲染失败则提示检查字段 |
| landing_note | `published` | 打开公开访问页 | 可复制链接分享 | 读取并展示内容 | 内容缺失则显示空态并返回编辑页 |

## 系统边界与 Handoff 契约

```text
HANDOFF: Editor -> Preview
  TRIGGER: 用户输入或点击预览
  PAYLOAD: { title, problem, solution, cta_text }
  SUCCESS: { preview_ready: true }
  FAILURE: { code: "INVALID_INPUT", message: string, retryable: true }
  TIMEOUT: 即时
  ON_FAILURE: 提示用户补全字段

HANDOFF: Storage -> Public Page
  TRIGGER: 打开公开页
  PAYLOAD: { landing_note }
  SUCCESS: { title, problem, solution, cta_text }
  FAILURE: { code: "NOT_FOUND", message: string, retryable: false }
  TIMEOUT: 即时
  ON_FAILURE: 显示内容不存在
```

## 客户端与服务端边界

- 哪些逻辑必须在服务端：本期无强制服务端逻辑
- 哪些数据不能信任前端传入：本期不涉及多用户场景
- 哪些接口必须鉴权：本期无
- 前端确认后才能冻结哪些接口契约：若后续加后端，只冻结内容读取结构
- 哪些操作必须幂等：内容保存
- 哪些流程需要后台异步处理：本期无
- 哪些逻辑允许前端做乐观更新，哪些不允许：全部以前端即时更新为主

## 认证、授权与安全

- 认证方式：本期无
- 会话或 Token 生命周期：不适用
- 权限模型：不适用
- 哪些接口需要限流：不适用
- 密钥与凭据放在哪里：本期无

## API 设计约束

- 返回格式：本期默认无 API
- 错误码约定：若有本地错误，使用简单错误常量即可
- 分页约定：不适用
- 幂等键或去重策略：保存动作可覆盖同一份草稿
- 超时约定：前端操作应近乎即时
- 版本策略：不适用

## 状态管理方案

- 全局状态：当前页面内容
- 页面局部状态：输入框状态、校验状态
- 缓存策略：localStorage 或本地文件
- 接口缓存失效条件：保存后立即覆盖旧内容
- 乐观更新与回滚策略：输入立即更新；保存失败时提示重试

## 非功能约束

- 性能上限：页面应在普通本地环境快速打开
- 成本上限：默认零额外服务成本
- 安全底线：不存放密钥，不暴露私人敏感信息
- 可用性目标：自己可稳定访问即可
- 可维护性目标：一小时内能改完一个核心字段或页面结构
- 观测性目标：最小 console 日志足够定位问题

## 合规与数据治理

- 数据是否包含个人信息：默认无
- 是否支持用户删除数据：可直接清空本地草稿
- 是否有地域合规要求：无
- 数据保留周期：按个人需要

## 监控与排错

- 日志位置：浏览器控制台
- 关键指标：页面可访问、CTA 可点击、内容未丢失
- 报错排查原则：先看表单输入，再看本地存储，再看路由

## 测试与验证策略

| 验证类型 | 入口/命令 | 执行环境 | 预期结果 | 触发时机 | 失败后处理 |
| --- | --- | --- | --- | --- | --- |
| lint | `npm run lint` | 本地 | 无明显语法和规范错误 | 页面可用后 | 修复后重跑 |
| typecheck | `npm run typecheck` | 本地 | 类型通过 | 增加字段或页面后 | 修复类型问题 |
| manual flow | 打开编辑页并完成一轮输入、预览、公开访问 | 本地 | 三步闭环可走通 | 每次切片完成后 | 记录阻塞点并修复 |
| smoke test | 用隐身窗口打开公开页 | 本地或部署环境 | 页面可访问且内容正确 | 分享前 | 失败则暂不分享 |

## 发布与回滚

- 发布方式：部署到静态站点或本地运行给朋友看
- 回滚方式：回到上一个静态版本或恢复上一份内容草稿
- 稳定版本判定标准：自己和 1 到 2 个朋友都能打开并看懂页面
- 发布前检查清单：页面可访问、文案完整、CTA 可点击
- 发布后重点观察指标：访问次数、反馈数量、是否有人理解页面要表达什么

- 假设 2：手动收集反馈已经足够，不必先做自动化采集

### 主要风险与缓解

| 风险 | 触发信号 | 影响 | 缓解方案 |
| --- | --- | --- | --- |
| 过度设计 | 开始出现复杂权限、后台和抽象层 | 验证节奏被拖慢 | 回到“单人、单页、单流程”原则 |
| 页面太弱无法表达价值 | 朋友看不懂页面在说什么 | 得不到有效反馈 | 先强化标题、问题和 CTA 文案 |
