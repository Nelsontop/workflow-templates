# Minimal SaaS Example ARCH

## 生成约定

- 本示例用于演示如何把一个轻量 SaaS 的需求落成可执行架构文档
- 目标不是追求“最先进”架构，而是体现 trade-off、工作流、验证与回滚闭环
- 该示例遵循前端先行、后端随后进入的默认协作方式
- 如果一个工作流存在于代码、部署脚本或数据生命周期中，就应在本文件中留下入口、状态和失败恢复说明

## 技术目标

- 当前阶段：内部测试版，面向小范围团队试用
- 优先级：先打通“登录 -> 提交反馈 -> 查看列表/详情 -> 更新状态”的闭环，再考虑扩展能力
- 主要技术风险：登录方案未最终敲定、状态更新权限边界可能反复调整、用户量增加后 SQLite 的并发写入能力有限
- 本期主要服务的 PRD 目标：统一反馈入口、缩短首次确认时间、降低人工整理成本
- 本期明确不支撑的 PRD 非目标：评论系统、附件上传、SSO、移动端深度优化

## 用户范围与规模

- 自用 / 小范围内测 / 公开上线：小范围内测
- 预估并发用户数：同时在线 10 到 30 人
- 预估数据规模：首月 300 到 1000 条反馈
- 峰值场景或流量假设：版本发布日和试用日会出现短时集中提交

## 架构选型与取舍

### 总体架构模式

- 推荐模式：前后端分离 + 模块化单体后端
- 为什么选它：前端可以先独立确认页面和交互，后端用单服务即可满足当前规模，便于快速迭代
- 当前阶段不选其他模式的原因：微服务和事件驱动会增加协作和运维复杂度，当前业务边界还不值得拆分
- 未来可能触发演进的信号：反馈量明显上升、需要异步通知、引入更多角色和复杂权限

### 关键技术决策（ADR）

| 决策项 | 可选方案 | 当前选择 | 选择原因 | 代价/放弃了什么 | 何时需要重审 |
| --- | --- | --- | --- | --- | --- |
| 本地启动与联调 | 裸机脚本 / Docker Compose | Docker Compose | 降低团队环境差异，便于前后端统一启动 | 启动成本更高，容器日志排查稍复杂 | 团队反馈本地启动明显变慢时 |
| 前端架构 | 页面直连接口 / 页面 + 服务层 + 状态层 | 页面 + 服务层 + 状态层 | 便于后续替换占位接口并稳定状态管理 | 初期样板代码更多 | 页面数量很少且状态极简单时 |
| 后端架构 | 单体 API / 多服务拆分 | 单体 API | 当前团队小、需求边界未稳定，单体更易维护 | 模块间边界需要自律维护 | 出现独立扩容或部署需求时 |
| 数据库 | SQLite / PostgreSQL | SQLite | 内测阶段部署和维护最轻 | 并发与运维能力有限 | 并发写入增加或需要复杂查询时 |
| 登录方式 | 密码 / 邮箱验证码 | 邮箱验证码 | 内部试用场景下更易分发和回收账号 | 依赖邮件发送能力 | 邮件到达率不稳定或扩展外部用户时 |

## 技术栈

- 客户端：React + TypeScript + Vite
- 服务端：Node.js + TypeScript + Express 或等价轻量 HTTP 框架
- 数据库：SQLite
- 部署平台：Docker Compose
- 第三方服务：邮件发送服务，用于邮箱验证码登录
- 配置与密钥管理：服务端环境变量，前端只读取公开配置
- 异步任务/队列（如有）：本期无独立队列，验证码发送采用应用内异步调用
- 文件存储（如有）：本期无

## 运行时拓扑与部署约束

- 运行进程/容器：`web`、`server` 两个主要容器
- 服务间依赖顺序：`server` 先就绪，`web` 再启动；SQLite 文件由 `server` 管理
- 对外暴露端口：`web` 暴露给内部用户，`server` 仅供 `web` 和测试工具访问
- 内部网络与服务发现方式：Docker Compose 默认网络，前端通过服务名访问后端
- 持久化卷或数据目录：SQLite 数据文件挂载持久化卷
- 健康检查方式：`web` 检查首页可达，`server` 检查 `/health` 返回 200
- 环境变量分层：本地 / 测试 / 生产
- 备份与恢复策略：测试环境每日导出 SQLite 文件；必要时人工恢复
- 第三方服务故障时的降级策略：登录邮件发送失败时停止登录流程并提示稍后再试，不允许无验证登录

```text
│   │   ├── src/
│   │   │   ├── pages/
│   │   │   ├── features/
│   │   │   ├── services/
│   │   │   └── state/
│   └── server/
│       ├── src/
│       │   ├── features/
│       │   ├── middleware/
│       │   └── db/
└── reference/
    └── examples/
```

分层规则：

- 本项目采用：`feature-first` + 明确服务层边界
- 选择原因：按功能组织更利于小团队理解整条业务链路
- 允许例外的条件：鉴权、中间件、数据库访问等横切能力可放独立目录
- `apps/web` 中的界面层只处理展示和交互
- `apps/web` 中的状态层只处理状态流转
- `apps/web` 中的服务层只处理接口通信
- `apps/server` 中按 feature-first 分层，每个 feature 再拆 route/service/repository
- 业务规则不要直接散落在页面组件里，也不要混进数据库访问层

## 模块边界

| 模块 | 职责 | 不负责什么 |
| --- | --- | --- |
| auth | 登录、会话校验、用户身份恢复 | 反馈业务读写 |
| feedback | 反馈创建、查询、状态变更 | 登录鉴权 |
| audit | 记录关键操作日志 | 业务授权决策 |

边界检查：

- 哪些模块允许直接依赖数据库层：`auth`、`feedback`、`audit` 的 repository
- 哪些模块只能通过服务接口调用：前端页面、后端 route 层
- 哪些共享能力可以沉淀到 `reference/`：API 返回格式、错误码约定、表单校验规则
- 哪些能力暂时不抽象，避免过早复用：评论、通知、标签体系

## 数据模型

| 实体 | 关键字段 | 关系 |
| --- | --- | --- |
| user | `id`, `email`, `role`, `status` | 一个用户可提交多条反馈 |
| feedback | `id`, `title`, `description`, `type`, `status`, `created_by` | 归属于一个提交人 |
| audit_log | `id`, `actor_id`, `action`, `target_id`, `created_at` | 记录登录和状态更新等关键操作 |

补充要求：

- 写清主键策略、唯一约束、软删/硬删策略：使用 UUID 主键；`user.email` 唯一；本期不做软删
- 写清哪些字段由系统生成，哪些字段允许用户输入：`id`、时间戳、`created_by`、`status` 默认值由系统生成；标题、描述、类型由用户输入
- 若有状态字段，需与“状态与失败恢复矩阵”一致

## 核心工作流清单

| 工作流 | 触发入口 | 主要参与方 | 成功结果 | 失败结果 | 当前状态 |
| --- | --- | --- | --- | --- | --- |
| 用户登录 | 登录页提交邮箱验证码 | 用户、Web、Server、邮件服务 | 建立登录态并进入列表页 | 登录失败并提示重试 | Draft |
| 提交反馈 | 列表页点击“提交反馈”并提交表单 | 用户、Web、Server、DB | 创建反馈并返回列表顶部可见 | 返回校验或服务错误，不清空输入 | Draft |
| 查看反馈列表与详情 | 列表页加载或点击详情 | 用户、Web、Server、DB | 返回列表或详情数据 | 页面显示错态或详情错误提示 | Draft |
| 更新反馈状态 | 详情抽屉修改状态 | 产品/工程、Web、Server、DB、Audit | 状态更新成功并写入审计日志 | 回滚界面状态并提示失败原因 | Draft |
| 测试环境部署 | Docker Compose 启动 | 工程师、Docker、Web、Server、SQLite | 两个服务成功启动且健康检查通过 | 启动失败并阻塞提测 | Draft |

## 工作流状态与失败恢复矩阵

| 工作流/实体 | 状态 | 进入条件 | 用户可见表现 | 系统动作 | 失败/超时后的恢复方式 |
| --- | --- | --- | --- | --- | --- |
| 登录流程 | `requesting_code` | 用户提交邮箱 | 按钮进入发送中态 | 校验邮箱并请求发送验证码 | 超时后提示重试，不创建会话 |
| 登录流程 | `authenticated` | 验证码校验通过 | 跳转列表页 | 建立会话并记录登录日志 | 会话建立失败则返回登录页并提示重试 |
| feedback | `todo` | 创建反馈成功 | 列表显示“待处理” | 写入数据库 | 创建失败则保留表单输入 |
| feedback | `in_progress` | 有权限用户更新状态 | 详情与列表显示“处理中” | 更新状态并写审计日志 | 若并发冲突，提示刷新后重试 |
| feedback | `done` | 问题处理完成 | 显示“已完成” | 更新状态并保留时间戳 | 审计写入失败则整体视为失败并回滚更新 |

## 系统边界与 Handoff 契约

```text
HANDOFF: Web -> Server
  TRIGGER: 用户提交登录、创建反馈或状态更新
  PAYLOAD: { request_id, auth_token?, body }
  SUCCESS: { ok: true, data, message? }
  FAILURE: { ok: false, code, message, retryable }
  TIMEOUT: 5s
  ON_FAILURE: Web 显示明确错误，不清空可恢复输入

HANDOFF: Server -> DB
  TRIGGER: 业务服务创建、查询或更新反馈
  PAYLOAD: { sql_or_orm_call, params }
  SUCCESS: { row_count, rows? }
  FAILURE: { code, message, retryable: false }
  TIMEOUT: 2s
  ON_FAILURE: 返回服务错误，必要时回滚事务

HANDOFF: Server -> Mail Provider
  TRIGGER: 请求发送验证码
  PAYLOAD: { email, code, template_id }
  SUCCESS: { provider_message_id }
  FAILURE: { code, message, retryable: true }
  TIMEOUT: 10s
  ON_FAILURE: 不创建会话，提示稍后再试，并记录错误日志
```

## 客户端与服务端边界

- 哪些逻辑必须在服务端：用户身份校验、验证码校验、反馈写入、状态更新权限校验、审计日志记录
- 哪些数据不能信任前端传入：用户身份、角色、反馈状态变更权限、创建时间
- 哪些接口必须鉴权：反馈列表、详情、创建、状态更新接口
- 前端确认后才能冻结哪些接口契约：登录返回结构、反馈列表分页结构、详情结构、状态更新错误结构
- 哪些操作必须幂等：验证码校验提交、状态更新请求
- 哪些流程需要后台异步处理：邮件验证码发送
- 哪些逻辑允许前端做乐观更新，哪些不允许：状态更新允许乐观展示但失败必须回滚；创建反馈不做乐观新增

## 认证、授权与安全

- 认证方式：邮箱验证码 + 服务端会话
- 会话或 Token 生命周期：登录会话 7 天，验证码 10 分钟有效
- 权限模型：基于角色；内部试用同事可提交和查看，产品/工程可更新状态
- 哪些接口需要限流：登录验证码发送、验证码校验、提交反馈接口
- 敏感字段如何脱敏：日志中邮箱脱敏显示，验证码不落日志
- 密钥与凭据放在哪里：仅服务端环境变量
- 是否记录审计日志：是，登录和状态更新记录到 `audit_log`
- 安全相关失败的默认响应方式：返回统一错误码，不泄露内部实现细节
- 第三方回调或 webhook 如何验签：本期无第三方回调

## API 设计约束

- 返回格式：统一 `{ ok, data, message, code? }`
- 错误码约定：按鉴权失败、校验失败、资源不存在、并发冲突、服务错误分类
- 分页约定：列表页默认第一页，支持 `page`、`page_size`
- 幂等键或去重策略：状态更新请求可使用 `request_id` 去重
- 超时约定：前端请求 5 秒超时，后端调用邮件服务 10 秒超时
- 版本策略：内测阶段不拆版本，统一 `/api`
- 向后兼容策略：前端占位契约冻结后，后端字段只能新增、不能无通知删除

## 状态管理方案

- 全局状态：登录态、当前筛选条件
- 页面局部状态：提交弹窗开关、详情抽屉开关、表单输入态、提交中态
- 缓存策略：列表短时缓存，详情按需拉取
- 接口缓存失效条件：创建反馈成功、状态更新成功后失效列表和详情缓存
- 乐观更新与回滚策略：状态更新允许乐观切换，失败后回滚到原状态

## 非功能约束

- 性能上限：列表页首屏接口在 1 秒内返回，详情接口在 500ms 内返回
- 成本上限：测试环境维持在低成本单机部署
- 安全底线：所有业务接口均需鉴权，敏感凭据不入前端
- 可用性目标：内测阶段工作日可用，非工作时段允许短暂维护
- 可维护性目标：前后端均可在一个工作日内定位核心流程问题
- 观测性目标：关键请求带 `request_id`，关键失败可从日志快速定位

## 合规与数据治理

- 数据是否包含个人信息：是，包含内部邮箱
- 是否支持用户删除数据：本期不对普通用户开放删除，管理员可人工处理
- 是否有地域合规要求：暂无
- 数据保留周期：内测期间保留 90 天后评估
- 导出/删除请求如何执行：管理员通过后台脚本手动导出或删除

## 监控与排错

- 日志位置：`server` 容器应用日志，`web` 控制台错误日志
- 关键指标：登录成功率、反馈提交成功率、列表加载成功率、状态更新成功率
- 报错排查原则：先确认入口请求，再检查后端日志和 DB 状态，最后确认第三方邮件服务
- Trace / Request ID 策略：每次请求生成或透传 `request_id`
- 告警阈值：登录或提交成功率连续 10 分钟低于 95% 触发告警
- 值班或人工介入方式：内测阶段由工程负责人手动处理

## 发布与回滚

- 发布方式：Docker Compose 部署到内部测试环境
- 回滚方式：回滚到上一个镜像版本，并恢复最近一次 SQLite 备份
- 稳定版本判定标准：登录、创建、列表、详情、状态更新五条主路径稳定运行 3 天
- 发布前检查清单：环境变量齐全、数据库备份完成、健康检查通过、核心流程回归通过
- 灰度/分批策略（如有）：先对内部 10 人名单开放
- 数据库或配置变更的回滚限制：若执行破坏性 schema 变更，需先导出备份再发布
- 发布后重点观察指标：登录成功率、反馈提交成功率、状态更新成功率、错误日志量

## 假设、未决项与风险跟踪

### 当前假设

- 假设 1：内测阶段用户规模足够小，SQLite 可满足读写需求
- 假设 2：邮件服务可稳定支持验证码发送

### 未决问题

| 问题 | 影响范围 | 何时必须决定 | 决定人 |
| --- | --- | --- | --- |
| 是否允许普通用户编辑自己提交的反馈 | 影响权限和状态模型 | 后端接口冻结前 | 产品负责人 |
| 是否需要把审计日志暴露给产品查看 | 影响 UI 和权限 | 第二阶段规划前 | 产品 + 工程 |

### 主要风险与缓解

| 风险 | 触发信号 | 影响 | 缓解方案 |
| --- | --- | --- | --- |
| 前端边界未稳定就进入后端 | 接口字段频繁改动 | 返工和测试成本升高 | 先冻结占位契约再进后端 |
| 邮件服务不稳定 | 验证码发送失败率升高 | 登录链路不可用 | 增加重试与人工白名单兜底 |
| SQLite 写入竞争增加 | 状态更新冲突增多 | 数据一致性风险 | 升级到 PostgreSQL 并加强事务控制 |
