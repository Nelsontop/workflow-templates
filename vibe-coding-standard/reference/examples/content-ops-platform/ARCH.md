# Content Ops Platform Example ARCH

## 生成约定

- 本示例用于演示“内容矩阵运营后台”类型项目的架构写法
- 示例重点是内容生命周期管理和发布任务流转，不是内容消费前台
- 平台发布接口在本期必须预留，且要支持本地技能、CLI 或脚本调用
- 本期不实现真实平台 API，但必须把适配边界、回填方式和失败恢复写清

## 技术目标

- 当前阶段：内部运营后台 MVP
- 优先级：先打通“内容草稿 -> 审核 -> 发布任务 -> 本地触发发布 -> 状态回填”的闭环
- 主要技术风险：内容状态和任务状态容易脱节，平台适配接口设计不稳会导致后续接入返工，回填错误会直接影响运营判断
- 本期主要服务的 PRD 目标：统一管理内容矩阵任务、提升发布状态透明度、减少人工同步成本
- 本期明确不支撑的 PRD 非目标：真实平台 API 接入、复杂审批配置化、自动生成内容、财务与投放系统

## 用户范围与规模

- 自用 / 小范围内测 / 公开上线：小范围内测
- 预估并发用户数：5 到 20 名运营相关角色
- 预估数据规模：每周数十到数百条内容和发布任务
- 峰值场景或流量假设：内容集中排期日和活动日会出现批量创建与批量发布

## 架构选型与取舍

### 总体架构模式

- 推荐模式：前后端分离 + 模块化单体后端 + 平台适配接口层
- 为什么选它：运营后台和发布适配职责不同，统一发布接口可以先稳定业务模型，再逐步接入真实平台
- 当前阶段不选其他模式的原因：多服务和事件总线会增加联调复杂度，当前更需要稳定流程而非独立扩容
- 未来可能触发演进的信号：平台数量明显增加、发布触发方式多样化、回填任务需要异步队列

### 关键技术决策（ADR）

| 决策项 | 可选方案 | 当前选择 | 选择原因 | 代价/放弃了什么 | 何时需要重审 |
| --- | --- | --- | --- | --- | --- |
| 后端形态 | 单体 / 多服务 | 模块化单体 | 小团队更易维护，状态模型集中 | 模块边界需要自律维护 | 平台适配明显独立扩展时 |
| 发布集成方式 | 业务服务直连平台 / 统一适配层 | 统一适配层 | 能先冻结业务任务模型和对接协议 | 初期多一层抽象 | 接入平台数量仍然只有 1 且长期不变时 |
| 平台调用方式 | 后端内嵌调用 / 本地 CLI / 技能封装 | 适配层 + 本地技能/CLI 调用 | 与现有本地技能工作流兼容，便于人工排查和手动重试 | 自动化程度低于直接 API 集成 | 需要全自动批量发布时 |
| 数据库存储 | SQLite / PostgreSQL | SQLite 起步 | 便于快速搭环境和验证流程 | 并发和查询能力有限 | 任务量和并发增长时 |

## 技术栈

- 客户端：React + TypeScript 后台应用
- 服务端：Node.js + TypeScript + 轻量 HTTP 框架
- 数据库：SQLite
- 部署平台：Docker Compose
- 第三方服务：本期无真实平台服务接入
- 配置与密钥管理：服务端环境变量，本地技能或 CLI 的平台凭据独立管理
- 异步任务/队列（如有）：本期可先无队列，以同步触发和手动重试为主

## 运行时拓扑与部署约束

- 运行进程/容器：`web`、`server`，以及可选的本地 `publisher` CLI/脚本执行环境
- 服务间依赖顺序：`server` 先就绪，`web` 再启动；本地发布调用独立于主服务运行
- 对外暴露端口：`web` 为运营入口，`server` 仅供前端和本地发布调用访问
- 内部网络与服务发现方式：Docker Compose 默认网络；CLI 通过本地命令或 HTTP 调用统一发布接口
- 持久化卷或数据目录：SQLite 数据文件挂载卷
- 健康检查方式：`web` 首页可达，`server` `/health` 返回 200
- 环境变量分层：本地 / 测试 / 生产；本地发布凭据不入前端
- 备份与恢复策略：每日导出 SQLite 文件；关键任务导出 CSV 备份
- 第三方服务故障时的降级策略：平台侧不可用时，任务标为失败或待重试，不阻塞后台继续管理内容

分层规则：

- 本项目采用：`feature-first` + 单独适配层
- 选择原因：业务流程和平台对接边界需要显式分开
- 允许例外的条件：鉴权、通用校验、数据库访问等横切能力可独立目录
- `apps/web` 的页面层处理后台操作流
- `apps/server/features` 处理内容、审核、任务等核心业务
- `apps/server/adapters` 处理平台发布接口与本地技能/CLI 对接
- `apps/server/workflows` 负责跨实体状态流转

## 模块边界

| 模块 | 职责 | 不负责什么 |
| --- | --- | --- |
| content | 内容草稿、内容状态流转 | 平台发布执行 |
| review | 提交审核、通过、打回 | 真实平台接口 |
| publish_task | 生成和管理平台维度任务 | 内容正文编辑 |
| publisher_adapter | 统一发布接口、调用本地技能/CLI、回填结果 | 审核判断 |
| analytics | 基础状态聚合 | 深度投放分析 |

边界检查：

- 哪些模块允许直接依赖数据库层：各模块 repository
- 哪些模块只能通过服务接口调用：前端页面、CLI 触发入口
- 哪些共享能力可以沉淀到 `reference/`：错误码、发布任务字段约定、状态枚举命名
- 哪些能力暂时不抽象，避免过早复用：素材库、预算系统、自动分发策略

## 数据模型

| 实体 | 关键字段 | 关系 |
| --- | --- | --- |
| content_item | `id`, `title`, `body`, `status`, `owner_id` | 一个内容可生成多个发布任务 |
| publish_plan | `id`, `content_id`, `channels`, `scheduled_at` | 对应一次发布计划 |
| publish_task | `id`, `content_id`, `channel`, `account_id`, `status` | 每个平台/账号一条任务 |
| publish_attempt | `id`, `task_id`, `trigger_source`, `result_code`, `result_message` | 记录每次触发和回填结果 |

补充要求：

- 状态字段必须统一：内容状态和任务状态不能各自随意命名
- 任务必须保留最近一次触发时间、触发来源和失败原因
- 平台返回的外部 ID 与 URL 应预留字段，但允许为空

## 核心工作流清单

| 工作流 | 触发入口 | 主要参与方 | 成功结果 | 失败结果 | 当前状态 |
| --- | --- | --- | --- | --- | --- |
| 创建内容草稿 | 内容编辑页保存 | 运营、Web、Server、DB | 内容保存为草稿 | 返回校验或保存错误 | Draft |
| 提交审核与审核通过 | 审核页操作 | 运营、审核负责人、Web、Server、DB | 内容进入已通过状态 | 内容保持原状态并提示原因 | Draft |
| 生成发布任务 | 审核通过事件 | Server、DB | 按平台/账号生成待发布任务 | 任务生成失败并标记异常 | Draft |
| 触发发布 | 任务页按钮、本地 CLI、技能调用 | 运营、CLI/技能、Server、适配层 | 调用统一发布接口成功发起一次尝试 | 任务进入失败或待重试 | Draft |
| 回填发布结果 | 适配层返回结果 | 适配层、Server、DB | 任务状态更新，结果可查看 | 回填失败并记录人工处理信号 | Draft |
| 查看基础复盘 | 数据概览页 | 内容负责人、Web、Server、DB | 可看到状态分布和异常任务 | 显示部分数据缺失提示 | Draft |

## 工作流状态与失败恢复矩阵

| 工作流/实体 | 状态 | 进入条件 | 用户可见表现 | 系统动作 | 失败/超时后的恢复方式 |
| --- | --- | --- | --- | --- | --- |
| content_item | `draft` | 草稿保存成功 | 内容出现在草稿列表 | 写入内容主表 | 保存失败则保留输入内容 |
| content_item | `in_review` | 提交审核 | 出现在待审列表 | 更新状态时间戳 | 状态更新失败则仍停留草稿 |
| content_item | `approved` | 审核通过 | 可进入任务生成链路 | 触发发布任务生成 | 若任务生成失败，标记异常并提示人工排查 |
| publish_task | `pending` | 任务生成成功 | 出现在待发布列表 | 等待触发 | 可手动重试生成 |
| publish_task | `publishing` | 触发发布成功 | 任务显示“发布中” | 创建一次 `publish_attempt` | 超时则转为待重试 |
| publish_task | `success` | 回填成功 | 显示平台结果和外部 ID | 更新任务完成时间 | 不需恢复 |
| publish_task | `failed` | 调用或回填失败 | 显示失败原因 | 记录失败日志和最近尝试信息 | 可手动重试或标为人工处理 |
| publish_task | `retryable` | 适配层判定可重试失败 | 显示“待重试” | 保留失败原因和重试入口 | 手动或计划任务重试 |

## 系统边界与 Handoff 契约

```text
HANDOFF: Ops Console -> Publish API
  TRIGGER: 运营在任务页点击发布，或任务到达发布时间
  PAYLOAD: {
    task_id,
    channel,
    account_id,
    content_payload,
    media_refs?,
    scheduled_at?,
    trigger_source
  }
  SUCCESS: { accepted: true, attempt_id, status: "publishing" }
  FAILURE: { accepted: false, code, message, retryable }
  TIMEOUT: 5s
  ON_FAILURE: 任务转为 failed 或 retryable

HANDOFF: Publish API -> Local Skill/CLI
  TRIGGER: 发布适配层收到统一发布请求
  PAYLOAD: {
    channel,
    account_context,
    content_payload,
    media_refs?,
    callback_context
  }
  SUCCESS: { external_id?, external_url?, status: "success" | "submitted" }
  FAILURE: { code, message, retryable }
  TIMEOUT: 60s
  ON_FAILURE: 写入 publish_attempt，并返回失败结果给业务层

HANDOFF: Publish Result -> Ops Console
  TRIGGER: 适配层同步返回或异步回填结果
  PAYLOAD: {
    task_id,
    attempt_id,
    final_status,
    external_id?,
    external_url?,
    result_message?
  }
  SUCCESS: { updated: true }
  FAILURE: { updated: false, code, message, retryable: false }
  TIMEOUT: 5s
  ON_FAILURE: 标记人工处理并阻止状态假成功
```

要求：

- 统一发布接口必须先于真实平台接入被冻结
- 统一发布接口必须支持本地技能、CLI 或脚本调用
- 结果回填不得只靠页面轮询推断，必须有显式结果写入

## 客户端与服务端边界

- 哪些逻辑必须在服务端：状态流转、任务生成、统一发布接口、结果回填、基础聚合
- 哪些数据不能信任前端传入：任务状态、审核结果、账号上下文、外部发布结果
- 哪些接口必须鉴权：内容编辑、审核操作、任务触发、结果查看
- 前端确认后才能冻结哪些接口契约：内容详情结构、任务结构、统一发布接口入参、回填结果结构
- 哪些操作必须幂等：任务生成、发布触发、结果回填
- 哪些流程需要后台异步处理：平台发布触发和可能的异步回填
- 哪些逻辑允许前端做乐观更新，哪些不允许：审核按钮可短暂 loading，但任务和发布状态不得纯前端乐观假设成功

## 认证、授权与安全

- 认证方式：后台账号登录
- 会话或 Token 生命周期：短会话或后台 Session
- 权限模型：运营可编辑内容和触发发布，审核负责人可审核，负责人可看全局概览
- 哪些接口需要限流：发布触发接口、结果回填接口
- 敏感字段如何脱敏：账号标识和平台凭据不在前端透出
- 密钥与凭据放在哪里：仅本地技能/CLI 或服务端安全配置中
- 是否记录审计日志：是，审核和发布触发必须有日志
- 安全相关失败的默认响应方式：返回统一错误结构，不暴露平台凭据

## API 设计约束

- 返回格式：统一 `{ ok, data, message, code? }`
- 错误码约定：审核失败、任务冲突、发布失败、回填失败、权限不足分别编码
- 分页约定：内容列表和任务列表均支持分页
- 幂等键或去重策略：发布触发使用 `task_id + attempt_no` 或请求 ID 去重
- 超时约定：运营后台接口 5 秒内返回，平台调用允许更长超时
- 版本策略：统一 `/api`
- 向后兼容策略：统一发布接口的核心字段一旦冻结，不做破坏性修改

## 状态管理方案

- 全局状态：当前筛选条件、当前账号/平台上下文
- 页面局部状态：编辑表单、审核抽屉、发布按钮 loading 状态
- 缓存策略：列表短时缓存，任务详情按需刷新
- 接口缓存失效条件：审核通过、任务生成、发布触发、回填成功后失效相关缓存
- 乐观更新与回滚策略：草稿保存可局部乐观，审核和发布状态以服务端真实结果为准

## 非功能约束

- 性能上限：运营列表和任务列表在常规数据量下 1 秒内返回
- 成本上限：内部测试环境低成本运行
- 安全底线：平台凭据不进入前端，统一发布接口必须可审计
- 可用性目标：工作时间稳定支持运营操作
- 可维护性目标：新增一个平台适配器时不改动核心业务状态模型
- 观测性目标：每次发布尝试都能从任务 ID 追踪到触发来源和最终结果

## 合规与数据治理

- 数据是否包含个人信息：可能包含运营账号标识
- 是否支持用户删除数据：本期以后台人工处理为主
- 是否有地域合规要求：根据平台运营地区后续补充
- 数据保留周期：至少保留当前运营周期内数据
- 导出/删除请求如何执行：通过后台导出和管理员脚本

## 监控与排错

- 日志位置：`server` 日志、本地发布 CLI/技能执行日志
- 关键指标：任务创建成功率、审核通过率、发布触发成功率、结果回填成功率
- 报错排查原则：先看任务状态，再查 `publish_attempt`，最后看本地技能/CLI 日志
- Trace / Request ID 策略：每次任务触发和回填都带唯一请求 ID
- 告警阈值：发布失败率或回填失败率连续上升时告警
- 值班或人工介入方式：本期由运营负责人和工程负责人手动介入

| --- | --- | --- | --- | --- | --- |

| regression | 按 PRD 主流程手测：草稿 -> 审核 -> 任务 -> 触发发布 -> 回填 | 测试环境 | 主链路可走通 | 提交前 / 发布前 | 阻塞发布 |

## 发布与回滚

- 发布方式：Docker Compose 部署到内部运营测试环境
- 回滚方式：回滚前后端镜像，并恢复最近一次 SQLite 备份
- 稳定版本判定标准：主链路连续数日可稳定走通，失败任务可被正确重试或标记人工处理
- 发布前检查清单：环境变量齐全、统一发布接口契约已确认、本地技能/CLI 能完成一次演练、数据库已备份
- 灰度/分批策略（如有）：先对单个运营小组开放
- 数据库或配置变更的回滚限制：状态枚举和任务结构改动前必须确认迁移策略
- 发布后重点观察指标：任务创建成功率、发布触发成功率、回填成功率、失败任务占比

## 假设、未决项与风险跟踪

### 当前假设

- 假设 1：本地技能/CLI 能在运营流程里承担首版发布触发职责
- 假设 2：首版只需基础状态回填，不必先做深度数据采集

### 未决问题

| 问题 | 影响范围 | 何时必须决定 | 决定人 |
| --- | --- | --- | --- |
| 发布适配层最终以 HTTP、脚本还是队列方式承接 | 影响接口稳定性和部署形态 | 开始接真实平台前 | 工程负责人 |
| 平台任务失败后是否支持自动重试策略 | 影响任务模型和运营体验 | 第二阶段前 | 产品 + 工程 |

### 主要风险与缓解

| 风险 | 触发信号 | 影响 | 缓解方案 |
| --- | --- | --- | --- |
