# Content Ops Platform Example PRD

## 生成约定

- 本示例用于演示“内容矩阵运营后台”类型项目如何填写模板
- 示例强调内容运营主链路：AI 选题、AI 生成、AI 质检、自动排期、自动发布、状态回填、基础复盘
- 示例明确预留平台发布接口与 AI 工作流编排边界，但本期不实现真实平台 API 对接
- 未写入本文件的功能，默认不进入当前版本
- 关键用户流程必须能映射到同目录下 `ARCH.md` 的核心工作流与验证方式

## 产品概述

- 产品名称：Content Ops Console
- 一句话定义：一个帮助团队以 AI 自动化方式运营多账号、多平台内容矩阵的运营后台
- 目标用户：运营负责人、增长负责人、内容策略负责人
- 目标用户当前如何解决这个问题：通过表格、飞书群、平台后台和人工审稿串联流程，内容产能、发布节奏和异常处理都高度依赖人力

## 背景与问题

- 我们要解决的核心问题是什么：内容矩阵运营流程高度依赖人工选稿、审稿、复核和手动发布，导致产能受限、状态不透明、复盘滞后
- 这个问题为什么现在必须解决：账号数量和发布频次增加后，人工运营链路已经成为增长瓶颈，无法稳定支撑规模化内容矩阵
- 如果这次不做，代价是什么：发布效率继续受制于人工排班和审核节奏，人工同步成本高，异常处理和复盘缺少统一口径

### 证据

- 用户研究：运营同学反馈“每天都在挑内容、看草稿、盯发布和补状态”，重复劳动多，无法把时间放在策略上
- 行为数据：暂无统一系统数据，当前以表格和群消息为主
- 支持/销售/运营信号：日常运营复盘依赖人工拼接多个平台后台截图，异常任务需要靠群内提醒追踪
- 竞品或市场信号：成熟团队通常会建设自动化运营中台，把人工从主链路中移出，仅保留异常兜底

## 核心用户流程

### 流程 1：AI 自动生成内容任务

1. 系统按运营策略、账号计划和发布时间窗口触发内容生成批次
2. AI 自动生成标题、正文、素材建议、平台计划和发布时间建议
3. 系统保存内容任务并记录生成来源与置信度
4. 内容任务自动进入 AI 质检阶段

失败分支：

- 哪一步可能失败：生成失败、策略参数缺失、任务保存失败
- 失败时如何提示：任务进入异常队列并显示失败原因
- 用户下一步可以做什么：调整策略后重跑批次，或对单条任务人工接管

### 流程 2：AI 质检通过并生成发布任务

1. AI 质检模块检查内容完整性、平台规则、风险分数和发布时间合法性
2. 合格内容自动标记为可发布
3. 系统按平台和账号自动生成发布任务
4. 不合格或低置信度任务自动进入异常队列

失败分支：

- 哪一步可能失败：质检规则冲突、评分服务失败、任务生成失败
- 失败时如何提示：显示具体异常原因，避免误以为任务已进入发布
- 用户下一步可以做什么：人工接管、修改规则后重试，或联系技术排查

### 流程 3：自动触发发布并回填状态

1. 任务到达发布时间或触发自动发布窗口
2. 系统自动调用本地 CLI / 技能触发统一发布接口
3. 系统把任务交给平台发布适配接口
4. 发布结果回填为成功、失败或待重试

失败分支：

- 哪一步可能失败：触发命令失败、平台适配层返回错误、结果回填失败
- 失败时如何提示：任务状态显示失败原因和最近一次尝试时间
- 用户下一步可以做什么：人工接管重试、修改内容后重发、标记为终止任务

### 流程 4：处理异常任务

1. 运营负责人进入异常队列页
2. 查看高风险、低置信度、发布失败和回填失败任务
3. 选择重试、人工改写后重发或终止任务
4. 系统记录人工接管原因和最终结果

失败分支：

- 哪一步可能失败：异常原因不明确、任务锁定失败、二次发布失败
- 失败时如何提示：显示“需工程介入”或“等待重试窗口”
- 用户下一步可以做什么：刷新、升级处理或查看任务详情

### 流程 5：查看自动化复盘

1. 增长负责人进入数据概览页
2. 查看按平台和账号聚合的发布状态
3. 查看自动化完成率、人工介入率、成功数、失败数和待处理数
4. 对高异常来源做策略调整

失败分支：

- 哪一步可能失败：数据回填不完整、聚合失败、任务状态过期
- 失败时如何提示：显示“数据待同步”或“部分结果缺失”
- 用户下一步可以做什么：刷新、手动同步或查看异常详情

补充要求：

- 每条核心用户流程都应能映射到 `ARCH.md` 中至少一个核心工作流
- 若流程涉及等待、异步处理、外部服务或人工介入，需在 PRD 先写清用户预期结果
- 若流程失败后允许重试、撤销或回退，需写清用户可执行动作

## 数据与业务规则

| 实体 | 字段 | 业务含义 | 约束/校验 |
| --- | --- | --- | --- |
| content_item | `title` | 内容标题 | 必填，最多 80 字符 |
| content_item | `body` | 内容正文 | 必填 |
| content_item | `status` | 待质检/可发布/异常/已终止 | 系统维护的流转状态 |
| content_item | `confidence_score` | AI 生成与质检置信度 | 0 到 1 之间 |
| publish_task | `channel` | 平台标识 | 必填，枚举值 |
| publish_task | `account_id` | 发布账号 | 必填 |
| publish_task | `status` | 待发布/发布中/成功/失败/待重试 | 系统维护 |
| publish_result | `external_id` | 平台侧返回的内容 ID | 可为空，发布成功后回填 |
| exception_case | `reason_code` | 异常原因 | 必填，枚举值 |

补充规则：

- 哪些字段用户可编辑：策略参数、平台计划、发布时间窗口、异常任务的人工修订内容
- 哪些数据只读：任务状态、最近一次发布时间、平台返回结果、AI 评分和尝试记录
- 哪些规则会影响结果计算、统计或计费：任务状态决定是否计入自动完成、人工接管、成功、失败或待处理统计

## 发布与回滚

- 计划如何发布：先发布到内部运营测试环境
- 是否灰度：是，先由单个运营小组试用
- 触发回滚的条件是什么：自动生成链路不可用、任务状态错乱、统一发布接口无法触发、异常任务堆积、回填结果错误率过高
- 回滚后用户侧预期表现是什么：回到表格 + 人工选稿审稿 + 手工平台后台操作方式
- 回滚后哪些数据或状态可能受影响：已创建但未导出的任务记录需要人工比对，自动生成批次需人工补齐
- 发布后重点观察哪些指标：自动化任务创建成功率、人工介入率、发布触发成功率、结果回填成功率
